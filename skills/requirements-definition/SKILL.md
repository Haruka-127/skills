# 要件定義スキル

実装に着手する前に「何を作るのか」「なぜ作るのか」「どこまで作るのか」を明確にし、要件定義書としてMarkdownファイルに出力する。

このスキルの価値は「曖昧なアイデアや要望を、実装計画書を作成できるレベルの具体的な要件に落とし込むこと」にある。ユーザーとの対話を通じて、漏れや矛盾を早期に発見し、手戻りの少ない開発の土台を作る。

## 対象シナリオ

- **新規プロジェクト**: ゼロから何かを作る場合。全体像の把握と優先順位づけが重要
- **新機能追加**: 既存プロジェクトに機能を追加する場合。既存アーキテクチャとの整合性が重要
- **既存機能の改善**: パフォーマンス改善、リファクタリング、UX改善など。現状の問題の正確な把握が重要

## ワークフロー

### 1. 初期ヒアリング

ユーザーの依頼内容から、まず大枠を把握する。

把握すべき情報:
- 何を作りたいのか（ざっくりとした概要）
- どのシナリオに該当するか（新規プロジェクト / 新機能追加 / 既存改善）
- 背景や動機（なぜこれが必要なのか）

ユーザーの最初の依頼だけでは情報が不足するのが普通。この段階では完璧な理解を目指さず、次の調査と深掘りで補完する前提で進める。

### 2. コードベースの調査（既存プロジェクトの場合）

既存プロジェクトへの新機能追加や改善の場合、要件を正確に定義するために現状を把握する。

調査対象:
- **プロジェクト構成**: ディレクトリ構造、主要モジュール、技術スタック
- **関連する既存機能**: 変更・拡張の対象となりそうな領域
- **既存の制約**: CLAUDE.md、設定ファイル、アーキテクチャ上の制約
- **依存関係**: 外部ライブラリ、API、サービス連携

調査にはGlob、Grep、Readなどのツールを使い、実際のコードに基づいて判断する。

調査結果は会話の中だけで消費せず、要件定義書の「現状調査サマリ（既存プロジェクトの場合）」に要約して残す。後続の実装計画書作成時に前提を引き継げる状態にする。

新規プロジェクトの場合でも、テンプレートや参考プロジェクトが指定されていればそれを調査する。

### 3. 要件の深掘り（対話フェーズ）

ここがこのスキルの核心。ユーザーとの対話を通じて、以下の観点から要件を網羅的に洗い出す。

**対話の進め方:**
- 一度に大量の質問を投げない。関連する観点をまとめ、1回の質問で3〜5項目に絞る
- ユーザーの回答から新たな論点が見えたら、追加で掘り下げる
- 「決まっていない」「わからない」も有効な回答。無理に決めさせず、未決定事項として記録する
- コードベースの調査結果から推測できることは、確認の形で提示する（「現在のプロジェクトは〇〇の構成ですが、これに合わせる前提でよいですか？」）
- 対話は必要な回数だけ繰り返す。1回で終わることもあれば、複数回になることもある

#### 観点A: 目的・背景

- この要件の最終的なゴールは何か
- 誰のための機能か（エンドユーザー、開発者、運用担当者など）
- ビジネス上の動機や解決したい課題は何か
- 成功をどう測るか（定量的な指標があるか）

#### 観点B: 機能要件

- 具体的にどのような振る舞いを期待するか
- ユーザー（利用者）の操作フローはどうなるか
- 入力と出力は何か
- 主要なユースケースは何か
- エッジケースや例外的な状況でどう振る舞うべきか

#### 観点C: 非機能要件

- **パフォーマンス**: 応答時間、スループット、同時接続数などの要求
- **セキュリティ**: 認証・認可、データ保護、入力検証に関する要件
- **可用性**: 稼働率、障害時の振る舞い、復旧要件
- **スケーラビリティ**: 将来のデータ量やユーザー数の増加への対応
- **保守性**: ログ、監視、デバッグのしやすさ
- **互換性**: ブラウザ、OS、デバイス、APIバージョンなどの対応範囲

#### 観点D: 技術的制約

- 使用する技術スタック（言語、フレームワーク、インフラ）
- 既存アーキテクチャとの整合性要件
- 外部サービスやAPIとの連携要件
- 開発環境やCI/CDに関する制約

#### 観点E: データ要件

- 扱うデータの種類と構造
- データの保存先と永続化の方法
- 既存データからの移行の必要性
- データのライフサイクル（作成、更新、削除、アーカイブ）

#### 観点F: スコープと優先順位

- 今回のスコープに含めるもの / 含めないもの
- MVP（最小限の実装）はどこまでか
- フェーズ分けが必要か（Phase 1, Phase 2 ...）
- 優先順位の高い要件はどれか

#### 観点G: 受入基準

- 各要件が「完了」と言えるための具体的な条件
- テストで検証すべき振る舞い
- ユーザーが確認すべきポイント

**観点の取捨選択:**
すべてのプロジェクトですべての観点が必要なわけではない。シナリオに応じて重点を調整する。

- 新規プロジェクト → 全観点を広く確認。特に目的・スコープ・技術的制約を重点的に
- 新機能追加 → 機能要件・データ要件・既存との整合性を重点的に。技術スタックは既存に合わせる前提
- 既存改善 → 現状の問題の正確な把握、非機能要件、受入基準を重点的に

### 4. 要件の確認と合意

要件定義書を書き始める前に、ここまでに整理した内容を短く要約し、次の3つに分けて確認する。

- **確定事項**: ユーザーが明示的に合意した内容
- **仮置き事項**: 現時点では妥当だが、今後見直す可能性がある前提
- **未決定事項**: 後続で判断が必要な論点

推測を含む内容は、そのまま確定事項として書かない。必要なら確認質問を追加し、判断材料が足りない場合は未決定事項として残す。

### 5. 要件定義書の作成

対話で得られた情報を整理し、以下の構成で要件定義書を作成する。

## 要件定義書テンプレート

```markdown
# 要件定義書: [プロジェクト/機能名]

## 概要
[何を作るのか、1〜3文で簡潔に記述する]

## 背景・目的
[なぜこれが必要なのか。解決したい課題やビジネス上の動機を記述する]

## 対象ユーザー
- [誰がこの機能を使うのか。複数のユーザー種別がある場合は列挙する]

## シナリオ種別
[新規プロジェクト / 新機能追加 / 既存改善]

## 現状調査サマリ（既存プロジェクトの場合）
- [関連する既存機能や画面]
- [影響を受ける主要モジュールやファイル]
- [既存アーキテクチャ・規約・制約]
- [今回の要件定義で前提とした既存挙動]

## 機能要件

### FR-1: [機能名]
- **説明**: [この機能が何をするか]
- **ユースケース**: [どのような場面で使われるか]
- **入力**: [何を受け取るか]
- **出力**: [何を返す/表示するか]
- **受入基準**:
  - [ ] [具体的な完了条件1]
  - [ ] [具体的な完了条件2]

### FR-2: [機能名]
[同様の構造で列挙する]

## 非機能要件

### パフォーマンス
- [具体的な数値目標がある場合は記載する]

### セキュリティ
- [認証・認可、データ保護に関する要件]

### 可用性
- [稼働率、障害時の振る舞い]

### スケーラビリティ
- [将来の拡張に関する要件]

### 保守性
- [ログ、監視、運用に関する要件]

### 互換性
- [対応環境、APIバージョンなど]

## 技術的制約
- [技術スタック、アーキテクチャ上の制約を列挙する]

## データ要件
- [扱うデータの種類、保存方法、移行の必要性など]

## スコープ

### やること
- [今回のスコープに含める項目を列挙する]

### やらないこと
- [明示的にスコープ外とする項目を列挙する]

## 優先順位
| 要件ID | 要件名 | 優先度 |
|---|---|---|
| FR-1 | [機能名] | 高 |
| FR-2 | [機能名] | 中 |

## 未決定事項
- [対話の中で決まらなかった事項。後で判断が必要なものを列挙する]

## 用語集（必要な場合）
| 用語 | 定義 |
|---|---|
| [用語] | [その用語の意味] |
```

## 作成ルール

### ファイル名
要件の内容がわかるファイル名をつける。配置場所は既存のプロジェクト慣習を優先し、`requirements/` や `docs/` など要件定義書の保存先が定まっていればそれに従う。特に慣習が見当たらない場合は、プロジェクトのルートディレクトリをデフォルトとする。

例:
- `requirements-user-auth.md`
- `requirements-api-redesign.md`
- `requirements-performance-optimization.md`

### 対話の質

良い対話:
- ユーザーの回答を正確に反映し、勝手に解釈を広げない
- 調査結果に基づく具体的な選択肢を提示して判断を助ける
- 「決まっていない」を許容し、未決定事項として正直に記録する
- 重要な判断ポイントでは、選択肢ごとのトレードオフを提示する

悪い対話:
- 一度に10個以上の質問を投げてユーザーを圧倒する
- ユーザーが答えていないことを勝手に決める
- 抽象的な質問ばかりで具体的なイメージが湧かない
- コードベースを調べればわかることをユーザーに聞く

### 具体性のレベル

要件定義書を読むのは実装計画書を作成するエージェントである。曖昧な要望の羅列ではなく、実装計画に落とし込める精度が必要。ただし、要件定義の段階で未確定の設計や技術選定を勝手に固定しない。

良い例:
- 「ユーザーはメールアドレスとパスワードでログインできる」
- 「ログイン成功時には認証済みセッションを開始できる」
- 「認証失敗時には理由を過度に開示しないエラーメッセージを表示する」
- 「セッションの有効期限は業務要件に応じて設定可能である」

悪い例:
- 「ログイン機能を作る」
- 「JWTを使い、有効期限は24時間、bcryptで照合する」

技術方式が未決定の場合は、要件として断定せず「技術的制約」または「未決定事項」に記載する。

### 不要なセクションの省略
テンプレートのすべてのセクションを埋める必要はない。プロジェクトの規模や性質に応じて、不要なセクションは省略する。小さな機能追加に対して大規模プロジェクト級の要件定義書を作らない。

### 完了条件
要件定義書はチャットに貼るだけで終わらせず、必ずMarkdownファイルとして実際に作成する。作成後はファイルパスを明示し、次のステップ（実装計画書の作成）にスムーズに進める状態にする。

### 実装計画書との連携
作成した要件定義書は `implementation-plan` スキルの入力として使える。要件定義書を渡して「この要件で実装計画書を作成して」と依頼することで、要件の意図を正確に反映した実装計画書を作成できる。
