# コードレビュースキル

git diffで検出された変更を多角的にレビューし、問題点や改善点を重要度別に整理して報告する。

このスキルの目的は、コードの変更が安全で品質が高いことを確認すること。単にスタイルの好みを押し付けるのではなく、実際にバグを生む箇所、セキュリティリスク、保守性を損なう変更など、実害につながる問題を優先的に検出する。

## ワークフロー

### 1. レビュー対象の特定

以下の優先順でレビュー対象を決定する。

1. **ステージ済みの変更がある場合**: `git diff --staged` の出力のみをレビュー対象とする
2. **ステージ済みの変更がない場合**: `git diff` の出力（未ステージの変更）をレビュー対象とする。加えて `git status` で untracked files（新規作成されたファイル）を検出し、存在すればそのファイルの内容もレビュー対象に含める

変更もuntrackedファイルもない場合は、その旨をユーザーに伝えて終了する。

特定のファイルやコミット範囲をユーザーが指定した場合は、そちらを優先する。

### 2. 変更内容と周辺コンテキストの調査

diffだけを見て表面的に判断しない。変更の意図と影響を正確に理解するために、以下を確認する。

- **変更されたファイルの全体像**: 変更箇所だけでなく、そのファイルの役割や構造を把握する
- **関連コード**: 変更された関数・クラスの呼び出し元、依存先、テスト
- **プロジェクトの規約**: CLAUDE.md、lint設定、既存のコーディングパターン
- **最近のコミット履歴**: 変更の背景を理解するために直近のコミットメッセージを確認する

調査にはGlob、Grep、Readなどのツールを使い、推測ではなく実際のコードに基づいてレビューする。

### 3. 多角的レビュー

以下の観点から変更を検査する。すべてに無理やり指摘を作る必要はないが、各観点を確認したうえで判断する。

#### バグ・ロジックエラー
- off-by-oneエラー、null/undefined参照、型の不整合
- 条件分岐の漏れ、エッジケースの未処理
- 非同期処理の誤り（race condition、未awaitのPromiseなど）
- リソースリーク（未クローズのファイル、コネクションなど）

#### セキュリティ
- インジェクション脆弱性（SQL、XSS、コマンドインジェクション）
- 認証・認可のバイパス可能性
- 機密情報の露出（ハードコードされた秘密鍵、ログへの出力）
- 入力値の検証不足

#### パフォーマンス
- N+1クエリ、不要なループ内処理
- メモリの過剰使用（大きなデータの不要なコピーなど）
- キャッシュの無効化や非効率なアルゴリズム
- 不要な再レンダリングや再計算

#### 設計・アーキテクチャ
- 単一責任の原則からの逸脱
- 不適切な抽象化（過剰または不足）
- 既存のアーキテクチャパターンとの不整合
- モジュール間の不要な結合

#### 可読性・保守性
- 意図が伝わりにくい命名やロジック
- 過度に複雑な条件分岐やネスト
- マジックナンバーや説明のない定数
- 重複コードの導入

#### テスト
- 変更に対応するテストの有無
- テストが実際の振る舞いを検証しているか
- エッジケースのテストカバレッジ

#### 互換性・影響範囲
- 公開APIやインターフェースの破壊的変更
- 後方互換性への影響
- 他のモジュールやサービスへの波及

### 4. 指摘の整理と報告

指摘は重要度をつけて整理する。

- **致命的（Critical）**: そのままマージするとバグ、セキュリティ脆弱性、データ損失などの実害が発生する
- **重要（Major）**: 機能は動くが、保守性・パフォーマンス・設計に大きな問題がある
- **軽微（Minor）**: あるとより良くなる改善提案

## レビュー結果の出し方

```markdown
## レビュー概要
- [変更全体の要約と総合評価を1〜3文で]

## 指摘事項

### [致命的/重要/軽微] [短いタイトル]
- **ファイル**: `path/to/file.ts:行番号`
- **問題**: [何が問題か]
- **根拠**: [なぜそれが問題なのか、どういう状況で実害が出るか]
- **提案**: [具体的な修正方法]

## 良い点
- [変更の中で良いと感じた点があれば挙げる]

## 総合判断
- [そのままマージしてよい / 軽微な修正後マージ可 / 修正が必要]
```

指摘がゼロの場合も「問題なし」で終わらせず、確認した観点と安心してマージできる根拠を簡潔に述べる。

## レビュー品質の基準

良いレビュー:
- 指摘に根拠があり、実害が説明されている
- 修正方法が具体的で、レビューを受けた人がすぐ対応できる
- 重要な問題にフォーカスし、些末な指摘で埋もれさせない

避けるレビュー:
- コードベースを確認せず、diffだけを見て一般論を述べる
- 個人の好みをルールのように押し付ける
- 変更の意図を理解せずに表面的な指摘をする
- 軽微なスタイルの問題を大量に列挙する

## 例

**悪い指摘:**
- 「この変数名はもう少しわかりやすくしたほうがいいです」

**良い指摘:**
- 「`processData` 関数内の `items.filter(x => x.status).map(x => x.id)` で、`status` が falsy な値（0や空文字列）の場合に意図せずフィルタされます。`x.status !== null && x.status !== undefined` のように明示的に比較すると安全です」(`src/utils/process.ts:42`)
