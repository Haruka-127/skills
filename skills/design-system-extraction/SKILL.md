# デザインシステム抽出スキル

指定されたURLのWebページを取得・分析し、デザインシステムの構成要素（カラーパレット、タイポグラフィ、スペーシング、コンポーネントパターン）を抽出する。

このスキルの価値は「参考にしたいWebサイトのデザインを、自分のプロジェクトですぐに使えるデザイントークンとして構造化すること」にある。目視で色を拾ったりDevToolsで調べる手間を省き、一貫性のあるデザインシステムとして整理する。

## ワークフロー

### 1. URL入力と対象の確認

ユーザーからURLを受け取り、抽出の方針を確認する。

確認すべき情報:
- 対象URL（必須）
- 特に注目したいデザイン要素があるか（省略可。特定の色、フォント、コンポーネントなど）
- 抽出結果の用途（省略可。どのようなプロジェクトに適用するか）
- ダークモード対応の有無を調べるか（省略可）

URLが複数ある場合は、1つずつ順番に処理する。追加情報がなくても、URLが明示されていれば「全体的に抽出する」をデフォルトとして先に進める。

### 2. Webページの取得と分析

`webfetch` ツールを使い、対象URLのHTMLコンテンツを取得する。HTML構造の解析が目的なので、まずは `format: "html"` を使う。

#### 2a. HTMLの取得

```text
webfetch(url=<対象URL>, format="html")
```

取得したHTMLから以下を特定する:
- `<link rel="stylesheet">` で参照されている外部CSSファイルのURL
- `<style>` タグ内のインラインCSS
- 要素の `style` 属性で指定されたインラインスタイル
- CSS変数（カスタムプロパティ）の定義（`:root` や `html` セレクタ内）
- `<base href>` の有無

#### 2b. 外部CSSの取得

HTMLから特定した外部CSSファイルのURLに対して `webfetch` を実行し、CSSの内容を取得する。CSS本文は `format: "text"` を基本とする。

- 相対パスの場合は、元のURLと `<base href>` を考慮して絶対URLを構築する
- `//example.com/app.css` のような protocol-relative URL は元ページのスキームを補完する
- 主要なCSSファイル（メインのスタイルシート）を優先的に取得する
- `@import` があれば、主要なものを追加でたどる
- CDNから読み込まれるフレームワーク（Bootstrap、Tailwindなど）のCSSは、フレームワーク名とバージョンを記録するにとどめ、カスタムCSS部分を重点的に分析する
- CSSの取得に失敗した場合は、失敗したURLと影響範囲を備考に残し、取得できたHTML/CSSだけで分析を続ける

#### 2c. CSS変数の優先確認

多くのモダンなWebサイトはCSS変数でデザイントークンを管理している。まず以下を確認する:

- `:root` や `html` セレクタに定義されたCSS変数
- `data-theme` や `.dark` などのテーマ切り替え用の変数セット
- CSS変数が体系的に定義されている場合、それをデザインシステムの骨格として活用する

#### 2d. 取得失敗時のフォールバックと制約

- ブラウザ実行環境やDOMの hydration がない前提で進める。computed style や JavaScript 実行後にだけ現れるスタイルは取得できない場合がある
- CSS-in-JS や JavaScript 生成スタイルは、静的HTMLや取得済みCSSから確認できる範囲のみを対象にする
- 動的なテーマ切り替えや hover / active の見た目は、静的CSSに定義があるものだけを記録する
- 取得できなかった情報は推測で補完せず、未検出または不確実として残す

### 3. デザイン要素の抽出

取得したHTML/CSSから、以下の4カテゴリのデザイン要素を抽出する。

#### 3a. カラーパレット

抽出対象:
- **プライマリカラー**: ブランドを象徴する主要色（ロゴ、CTAボタン、リンクなどに使われる色）
- **セカンダリカラー**: 補助的に使われるアクセントカラー
- **ニュートラルカラー**: 背景色、ボーダー色、テキスト色などのグレースケール系
- **セマンティックカラー**: 成功（緑）、警告（黄）、エラー（赤）、情報（青）などの意味を持つ色
- **グラデーション**: 使用されているグラデーションの定義

抽出方法:
1. CSS変数から色の定義を抽出（`--color-*`, `--bg-*`, `--text-*` など）
2. CSSルールから `color`, `background-color`, `border-color` などの値を収集
3. 頻出する色の値を集計し、パレットとして整理する
4. 色は `hex`, `rgb`, `hsl` のうち元のCSSで使われている形式を記録し、hexも併記する

分類の判断基準:
- ヘッダー、フッター、ナビゲーション、CTAボタンに使われる色 → プライマリ候補
- ホバー状態やアクティブ状態で使われる色 → プライマリのバリエーション
- `body` や大きなコンテナの `background-color` → ニュートラル/背景色
- `body` や段落の `color` → ニュートラル/テキスト色

#### 3b. タイポグラフィ

抽出対象:
- **フォントファミリー**: 使用されているフォントの一覧（見出し用、本文用、コード用など）
- **フォントサイズ**: 見出し（h1〜h6）、本文、キャプション、ラベルなどのサイズ体系
- **フォントウェイト**: 使用されている太さのバリエーション
- **行間（line-height）**: テキストの行間設定
- **字間（letter-spacing）**: 字間の設定（特に見出しやボタン）
- **Webフォントの読み込み**: Google Fontsなどの外部フォントサービスのURL

抽出方法:
1. `@font-face` 定義や `<link>` タグでのフォント読み込みを特定
2. `font-family`, `font-size`, `font-weight`, `line-height`, `letter-spacing` の値を要素別に収集
3. 見出し（h1〜h6）、段落（p）、リンク（a）、ボタン（button）などの主要要素のタイポグラフィを整理
4. フォントサイズは元の単位（px, rem, em）を保持しつつ、px換算値も併記する

#### 3c. スペーシング・レイアウト

抽出対象:
- **スペーシングスケール**: margin, padding, gap に使われている値の体系
- **コンテナ幅**: max-width で定義されるコンテンツ幅
- **グリッドシステム**: カラム数、ガター幅、グリッドの定義
- **ブレークポイント**: メディアクエリで定義されるレスポンシブの切り替え点
- **ボーダー半径**: border-radius の値の体系
- **シャドウ**: box-shadow の定義

抽出方法:
1. CSS変数からスペーシング関連の値を抽出（`--spacing-*`, `--gap-*` など）
2. `margin`, `padding`, `gap` の値を収集し、頻出する値をスケールとして整理
3. `@media` クエリからブレークポイントを抽出
4. `max-width` の値からコンテナ幅を特定
5. `border-radius`, `box-shadow` の値を収集・分類

#### 3d. コンポーネントパターン

抽出対象:
- **ボタン**: バリエーション（primary, secondary, outline, ghost など）、サイズ、状態（hover, active, disabled）
- **フォーム要素**: input, select, textarea のスタイル
- **カード**: カードコンポーネントのスタイル（影、角丸、パディング）
- **ナビゲーション**: ヘッダー、サイドバー、パンくずリストのパターン
- **その他**: バッジ、タグ、アラート、モーダルなど特徴的なコンポーネント

抽出方法:
1. HTML構造からコンポーネントのパターンを特定（クラス名、セマンティックなHTML要素）
2. 各コンポーネントに適用されているCSSルールを集約
3. BEMやTailwindなどのCSS命名規則から、コンポーネントの意図を推測
4. 状態（hover, focus, active, disabled）のスタイル変化も記録

ユーティリティクラス主体のサイトでは、クラスの出現頻度だけで意味を決めない。HTML構造、ARIA属性、要素の役割、繰り返しパターンをあわせて判断する。

### 4. デザインシステムの構造化

抽出した要素を整理し、一貫したデザインシステムとしてまとめる。

整理のポイント:
- まず「直接抽出した値」を整理する。元のCSS変数名、セレクタ、ファイル名など、根拠を残せる場合は残す
- その上で、十分な根拠があるものだけを「正規化したトークン案」として別レイヤーで提示する
- 似た色を統合したり意味名を与えたりする場合は、どの要素・用途からそう判断したかを明記する
- フォントサイズやスペーシングは、複数箇所で反復している値だけをスケールとして整理する
- 判断が曖昧な項目は元の名前や値を保持し、`uncertain` / `low confidence` として扱う

### 5. 成果物の出力

原則として2つの成果物を用意する。ファイル書き込みが可能ならファイルとして作成し、書き込みできない環境では、そのまま保存できる内容を会話内に提示し、推奨ファイル名を明示する。

## 作成ルール

### ファイル名と配置

抽出結果は原則として2つの成果物としてまとめる。配置場所は既存のプロジェクト慣習を優先し、特に慣習がない場合はプロジェクトのルートディレクトリをデフォルトとする。

- ファイル作成が可能なら、実ファイルとして保存する
- ファイル作成ができない場合は、推奨ファイル名を添えて Markdown / CSS の内容をそのまま返す

- Markdownドキュメント: `design-system-[サイト名またはドメイン].md`
- CSSカスタムプロパティ: `design-tokens-[サイト名またはドメイン].css`

例:
- `design-system-stripe.md` / `design-tokens-stripe.css`
- `design-system-linear.md` / `design-tokens-linear.css`

### 色の正確性

- CSSから直接抽出した色をそのまま記録する。推測で色を補完しない
- 元のCSSでの表記（hex, rgb, hsl）を保持しつつ、hex値を併記する
- グラデーションは完全な定義を記録する
- 透明度が指定されている場合は rgba / hsla 形式で記録する

### 抽出と正規化の分離

- 直接抽出した値と、再利用しやすいように整理した正規化トークンを分けて記載する
- 元サイトに体系的なCSS変数がある場合は、その命名体系を優先して残す
- `primary`, `secondary`, `gray-100` などの意味名への再命名は、複数箇所の用途から裏付けられる場合のみ行う
- 判断が曖昧な場合は元の値や変数名を保持し、信頼度や備考で不確実性を示す
- 可能なら、元ファイル・セレクタ・変数名などの根拠を残す

### 命名規則

- CSS変数名はケバブケースを使用する（`--color-primary-light`）
- 色名には番号スケールを使用する（`gray-100` 〜 `gray-900`）
- 意味的な名前と数値スケールを組み合わせて使う
- 元サイトのCSS変数名が体系的に定義されている場合は、その命名体系を尊重する

### 不要なセクションの省略

テンプレートのすべてのセクションを埋める必要はない。抽出元のサイトに存在しない要素のセクションは省略する。検出できなかった要素を推測で埋めるのではなく、「検出なし」として正直に記録するか省略する。

### 抽出精度に関する注意

抽出結果には以下の制約がある。ドキュメントの「備考」セクションに該当する制約を記載する:

- JavaScriptで動的に生成されるスタイルは取得できない場合がある
- CSS-in-JSライブラリ（styled-components, Emotionなど）のスタイルはHTMLのインラインスタイルとしてしか見えないことがある
- ホバーや各種インタラクション状態のスタイルはCSSからは抽出できるが、実際の見た目と異なる場合がある
- レスポンシブデザインの全パターンは静的解析では完全には把握できない
- フォントの実際のレンダリングはOSやブラウザに依存する
- `<base href>`, `@import`, 外部CSSの取得制限によって、一部のスタイルを追跡しきれない場合がある
- ユーティリティクラス主体のサイトでは、頻出値だけでは意味を断定できない場合がある

### 既存スキルとの連携

抽出したデザインシステムは以下のスキルと連携して活用できる:

- `requirements-definition`: デザインシステムを要件の一部として参照する
- `implementation-plan`: デザイントークンの適用方法を実装計画に組み込む
- `implement`: CSSカスタムプロパティを実装時にそのまま使用する

### 完了条件

- Markdownドキュメント、または保存可能な Markdown 内容が提示されている
- CSSカスタムプロパティファイル、または保存可能な CSS 内容が提示されている
- ファイルを書き出した場合はファイルパスが明示され、書き出していない場合は推奨ファイル名が明示されている
- 抽出精度に関する注意事項が備考に記載されている
- ユーザーに対して、抽出結果の確認と必要に応じた調整を促している
