# 実装計画書レビュースキル

実装計画書を「これで本当に迷わず実装できるか」という観点でレビューする。

このスキルの目的は、文書の見た目を整えることではなく、計画の欠落・危険な前提・曖昧な指示を実装前に見つけることにある。とくにコーディングエージェント向けの計画書では、少しの曖昧さがそのまま誤実装や手戻りにつながるため、具体性と検証可能性を重視する。

## 基本方針

- まず計画書全体を読み、何を達成したい計画なのかを自分の言葉で要約してからレビューに入る。
- 推測だけで指摘しない。関連コード、設定、既存ドキュメントを確認し、計画の妥当性をコードベースに照らして判断する。
- 些末な書きぶりより、実装の成否や手戻りに効く指摘を優先する。
- 問題を指摘するだけで終わらせず、どう直せばよいかまで具体的に示す。
- 問題がない場合も「問題なし」で終わらせず、安心して実装に進める根拠を簡潔に述べる。

## ワークフロー

### 1. レビュー対象の特定

ユーザーが対象ファイルを明示していればそれを使う。明示されていなければ、会話中の文脈とリポジトリ内のファイル名から候補を探す。

優先候補:
- ユーザーが直前に作成した計画書
- `implementation-plan-*.md` のようなファイル
- `plans/` や `docs/` 配下の実装計画書

対象が複数あり得る場合は、調査結果を添えて確認する。候補が見つからない場合は、ユーザーにファイルパスを確認する。

### 2. 文書と周辺コンテキストの調査

Glob、Grep、Readなどのツールを使い、以下を必要な範囲で確認する。

- 計画書本文
- 依頼の背景となるユーザー要件
- 関連するソースコード、テスト、設定ファイル
- 開発ルールが書かれたファイル（例: `CLAUDE.md`, `README.md`, lint/test 設定）

確認時の観点:
- 計画書が想定しているファイルやモジュールは実在するか
- 既存アーキテクチャや命名規則と整合しているか
- すでに似た実装や制約が存在しないか

### 3. 多角的レビュー

最低でも次の観点をチェックする。全部に無理やり指摘を作る必要はないが、すべてを確認したうえで結論を出す。

#### 目的・スコープ
- 何を実現する計画かが明確か
- やること / やらないことの境界が十分か
- 目的に対して手段がずれていないか

#### 実現可能性・正確性
- 計画が既存コードの構造に沿っているか
- 指定された変更箇所や技術方針が現実的か
- 欠けている前提条件や移行作業がないか

#### 実装ステップの粒度・順序
- 各ステップが実装可能な粒度か
- 依存関係と実行順序が自然か
- 並列化できる箇所と逐次実行すべき箇所が整理されているか

#### 変更対象ファイルの妥当性
- 対象ファイルが足りているか、不要なものが混ざっていないか
- 新規作成 / 修正 / 削除の見立てが妥当か
- 実際に影響が出る関連ファイルが漏れていないか

#### リスク・移行・互換性
- 既存機能への影響が見えているか
- データ移行、設定変更、権限、パフォーマンス、後方互換性への配慮があるか
- ロールバックや段階的リリースが必要な変更なのに無視されていないか

#### テスト・検証計画
- テスト対象が目的と変更内容に対応しているか
- 正常系だけでなく境界条件や失敗系も見えているか
- 実装完了をどう確認するかが明確か

#### 要件トレーサビリティ・前提条件
- 要件定義書や関連資料がある場合、それが計画書に適切に引き継がれているか
- `要件対応表` などから、各要件が実装ステップとテストにどう対応するか追えるか
- `実装前提・未決定事項` が整理され、実装前に解消すべき論点と前提付きで進める論点が区別されているか
- 前提付きで進める場合、影響範囲や後で変更が入った場合のリスクが見えているか

#### エージェント実行可能性
- コーディングエージェントがそのまま着手できる具体性があるか
- 「改善する」「対応する」のような曖昧語が残っていないか
- 必要な判断材料が各ステップに埋め込まれているか

### 4. 指摘の整理

指摘は重要度をつけて整理する。

- **致命的（Critical）**: このまま実装すると高確率で誤実装・設計破綻・大きな手戻りにつながる
- **重要（Major）**: 実装はできるが、抜け漏れやリスクが大きく品質低下につながる
- **軽微（Minor）**: あるとより安全・明確になる改善点

各指摘には次の要素を入れる。

- **観点**: どの観点の問題か
- **問題**: 何が不足・矛盾・曖昧なのか
- **根拠**: 計画書またはコードベースのどこからそう判断したか
- **提案**: どう直すべきか

### 5. 必要なら改訂案まで示す

ユーザーがレビューだけでなく修正も求めている場合、重要な指摘から順に計画書の差し替え案を出す。全文を書き換えるより、まずは致命度の高い部分を直せる形で提示する。

## `implementation-plan` 由来の計画書で特に見る点

`implementation-plan` スキルで作られた計画書なら、次の項目が実質的に満たされているかを重点確認する。

- 概要が短くてもタスクの本質を言い表せているか
- 目的が技術作業の説明で終わらず、なぜ必要かまで触れているか
- 前提資料・既存コンテキストが、要件定義書や既存調査結果を正しく引き継いでいるか
- スコープの外側が明確で、過剰実装を防げるか
- 要件対応表から、重要な要件が実装ステップとテストへ落ちていることを確認できるか
- 実装前提・未決定事項が放置されず、扱い方が明記されているか
- 変更対象ファイルの表が実コードと整合しているか
- 実装ステップがコミット可能な粒度で、依存関係も妥当か
- リスク・注意点が現実の失敗パターンを捉えているか
- テスト方針が「何をどう確認するか」まで落ちているか

見出し名が多少違っても、同等の情報が入っていれば問題ない。形式より内容を優先する。

## レビュー結果の出し方

基本は次の形式で返す。

```markdown
## レビュー結論
- [全体評価を1〜3文で要約]

## 主要な指摘

### [致命的/重要/軽微] [短いタイトル]
- 観点: [例: テスト・検証計画]
- 問題: [何が問題か]
- 根拠: [計画書やコードベースの根拠]
- 提案: [どう直すか]

## 追加で確認するとよい点
- [ユーザーや作成者に確認すべき論点があれば列挙]

## 実装に進んでよいか
- [そのまま進める / 軽微修正後なら進める / 修正してから進める]
```

指摘がない観点は無理に節を立てなくてよい。ただし、重要な観点を確認済みであることは結論や指摘の中で伝える。

## レビュー品質の基準

良いレビュー:
- 計画書を読んだ実装担当が、そのまま修正に反映できる
- 根拠が具体的で、単なる好みの指摘になっていない
- 「何が抜けているか」だけでなく「なぜそれが危険か」がわかる

避けるレビュー:
- コードベースを見ずに一般論だけ述べる
- 曖昧な感想で終わる
- 軽微な表現の揺ればかりを大量に指摘する
- 実装可能性より文体の美しさを優先する

## 例

**悪い指摘:**
- 「少しわかりにくいです。詳細を書いたほうがよさそうです」

**良い指摘:**
- 「`src/api/users.ts` を変更対象に含めていますが、実際のルーティング定義は `src/server/routes/users.ts` にあります。現状の計画のままだと実装担当が誤った層を修正する可能性があります。ルート定義とハンドラ実装の両方を変更対象に明記してください」
